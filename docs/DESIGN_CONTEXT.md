## 1. ビジョン（何のためのデモか）

「ゴブリン討伐」シナリオの **見た目（グラフィックス）の方向性を、関係者で認識合わせするための 2D デモ**。
ゲームとして完成させることではなく、「この絵で行くか」を素早く見て・触って判断できる状態を保つことが目的。

## 2. 設計の柱（invariants／ブレさせない）

- **ファミコン／スーファミ風**のドット絵トーン。派手な現代エフェクトは足さない。
- **単一 HTML で配布**（ビルド不要・依存なし・ダブルクリックで動く）。素材は Base64 で `index.html` に同梱。
- **「知覚(perceptive)」コンセプト**: 戦う前の準備／情報収集（最大3つ）が戦闘を有利にする。これは派生元エンジンの
  `goblin` シナリオデータ（`design/scenario-goblin.yaml`）と一致させる。数値やボーナスを勝手に変えない。
- **視覚の一貫性**は `design/visual_bible.yaml`（キャラ・画風の固定記述子）と `design/image_prompts.yaml` に従う。
  新規画像を作るときは必ずここを参照（キャラの顔・衣装・画風が毎回ブレないため）。
- **派生元エンジンとの視覚整合**: 正準 LOD アセット `assets/reference/two_d/goblin/*/lod0〜3.png` を見た目のレンジ参照にする。

## 3. 非目標（やらないこと＝スコープ防衛）

- 派生元の決定論エンジン（`trpg_core`）の再実装・移植はしない（このリポジトリは見た目デモ専用）。
- ゲームバランスの作り込み、ストーリー拡張、セーブ機能などは範囲外（必要になったら §7 で明示的に決める）。
- 戦闘 RNG の決定論化はしない（デモは `Math.random()` のままでよい）。

## 4. 対象プロダクトの要点（index.html）

- 内部解像度 256×224 の単一 `<canvas>`、素の JavaScript。状態は `S`（探索/村/戦闘）、戦闘は `B`。
- シーン: `title|map|dialogue|battle|win|over`。判定は 2D6+修正（12=会心, 2=ファンブル）。
- **ズーム/LOD**: `S.lod`(0→1→2) を井戸周辺で上げると 16→32→64px にズーム。**詳細度+2 = 高精細アートを見せる場**。
- 詳しい実装マップは `CLAUDE.md`（Claude Code 用）を参照。設計チャットは通常コードの細部に立ち入らない。

## 5. 素材パイプライン（見た目を差し替える標準手順）

1. `design/visual_bible.yaml` / `image_prompts.yaml` に沿って画像生成（Nano Banana 等）→ `assets/source/` に置く。
2. `python tools/conform_hi.py` で透過・切出し → `assets/hi/atlas_hi.png` + `manifest_hi.json` を生成。
3. `index.html` の `SPR_HI`/`ATLAS_HI`（Base64）を更新（Claude Code が実施）。
4. VSCode でズーム+2して見た目確認（§10）。

## 6. 現在地（最新状態・ここを常に更新）

- ✅ リポジトリ作成・公開: https://github.com/piperendervt-glitch/perceptive-world-demo （public）
- ✅ 現バージョンのクリア可能性を確認（自動プレイで頭目撃破→「クリア！」到達、エラー0）。
- ✅ **詳細度+2 の高精細化を実装・検証済み**: `S.lod>=2` のとき `SPR_HI/ATLAS_HI` を参照。
  草/土/水/花/木/井戸/主人公が新素材に置き換わることを before/after で確認。
- ⏳ 未反映: この実装コミットを Claude Code が push（§8 の指示 P1）。
- ⏳ 既知の調整候補: (a) 地形タイルの継ぎ目（微小な縦線）を消す, (b) 主人公の上/横向きフレームが正面寄り（元画像由来）,
  (c) 高精細主人公が既存より横幅が広い（チビ体型）—採用可否は要判断。

## 8. Claude Code へのタスク指示（テンプレと現行キュー）

**テンプレ**（設計チャットはこの粒度で渡す。1タスク=1目的、検証条件を必ず書く）:
```
[目的] <何を達成したいか（1文）>
[背景] DESIGN_CONTEXT.md §<番号> / CLAUDE.md §<番号> を参照
[変更範囲] <触ってよいファイル/関数。それ以外は変えない>
[検証] <どうなれば完了か。スクショ/テストの合格条件>
[非目標] <やってほしくないこと>
```

**現行キュー**:
- **P1（適用と push）**: 「検証済みの詳細度+2 実装（`index.html` の SPR_HI/ATLAS_HI 追加、`tools/conform_hi.py`、
  `assets/hi/`、本 `docs/DESIGN_CONTEXT.md`）を取り込み、対象ファイルを明示 add してコミットし push。
  検証: ブラウザで井戸付近ズーム+2 時に新素材が表示され、コンソールエラー0。」
- **P2（任意・仕上げ）**: 「地形タイルの継ぎ目（縦線）を軽減。`tools/conform_hi.py` のタイル切出しで端1〜2px を
  さらにインセット、または隣接ピクセル複製で対処。検証: ズーム+2 で草/土の格子状の線が目立たない。」

## 9. 新しいチャットへの引き継ぎ手順（履歴が重くなったら）

1. 新しい設計チャットを開く。
2. **この `docs/DESIGN_CONTEXT.md` の全文を貼る**（リポジトリにあるので最新を取得）。
3. 最初の指示は「このコンテキストで設計を継続。§6 現在地と §8 現行キューから再開して」。
4. 以後も本文書を repo にコミットして更新し続ける（＝どのチャットからでも同じ真実を見る）。

> ポイント: 設計チャットの記憶は**この文書に外部化**されている。チャットは使い捨て可能で、文書が継続する。

## 10. 人間によるプレイテスト（VSCode）

`index.html` は単一ファイルで動くので、いちばん簡単なのはエクスプローラで **`index.html` をダブルクリック**して
既定ブラウザで開く方法。操作は 移動=矢印、決定/調べる=`Z`、キャンセル=`X`。井戸に近づくと右上の「＋拡大」で
詳細度+2 まで上げられる（＝高精細アートの確認）。

より開発向きには、VSCode 拡張 **Live Server / Live Preview** で `index.html` を開くと、保存のたびに自動リロードされて
イテレーションが速い。もしくは統合ターミナルで `python -m http.server 8000` を実行し、ブラウザで
`http://localhost:8000/index.html` を開く。ゲーム進行の目安は「村で準備を3つ集める→東の門から森→洞窟の奥で頭目を倒す→クリア」。
